## 4) Xcode Integration (App Target)

1. Add Sparkle via Swift Package Manager
   - Xcode > Add Package > `https://github.com/sparkle-project/Sparkle` (branch/tag for 2.x).
   - Add the `Sparkle` product to your app target (link and embed).

2. Embed SparkleÔÇÖs helper components
   - Ensure SparkleÔÇÖs XPC and helper app are copied into your app bundle:
     - Add a ÔÇťCopy FilesÔÇŁ build phase with Destination set to `Wrapper`, Subpath `Contents/Library/LaunchServices`, and include `Updater.app` from the Sparkle package products.
     - Add another ÔÇťCopy FilesÔÇŁ phase with Destination `XPC Services` and include `org.sparkle-project.InstallerLauncher.xpc`.
   - If using SPM, Xcode may add some phases automatically; verify both helper artifacts end up in the final `.app` bundle at the above locations.

3. App Sandbox & Entitlements (if sandboxed)
   - Enable App Sandbox and grant `Outgoing Connections (Client)` so Sparkle can reach your feed and downloads.
   - Hardened Runtime should be enabled; you generally donÔÇÖt need to disable library validation for Sparkle 2.

4. Info.plist keys
   - `SUFeedURL` (String): HTTPS URL to your appcast, e.g. `https://<your-domain>/appcast.xml`.
   - `SUPublicEDKey` (String): Base64 Ed25519 public key generated by Sparkle.
   - Optional defaults:
     - `SUEnableAutomaticChecks` (Boolean) default true
     - `SUAllowsAutomaticUpdates` (Boolean) default true
     - `SUShowReleaseNotes` (Boolean) per your preference
   - Ensure your feed and downloads are served over HTTPS. Avoid ATS exceptions if possible.

5. Swift integration (SwiftUI example)
   - Add an updater controller that lives for the lifetime of the app and a menu command to trigger updates.

   Example:

   ```swift
   import SwiftUI
   import Sparkle

   @main
   struct HardlinkerApp: App {
       @StateObject private var updaterController = SPUStandardUpdaterController(
           startingUpdater: true,
           updaterDelegate: nil,
           userDriverDelegate: nil
       )

       var body: some Scene {
           WindowGroup {
               ContentView()
           }
           .commands {
               CommandGroup(after: .appInfo) {
                   Button("Check for UpdatesÔÇŽ") {
                       // Either call on controller or direct on updater
                       updaterController.checkForUpdates(nil)
                       // or: updaterController.updater.checkForUpdates()
                   }
               }
           }
       }
   }
   ```

   - For advanced control, implement `SPUUpdaterDelegate` (e.g., to customize feeds, channels) and/or `SPUUserDriverDelegate`.


## 5) Keys, Signing, Notarization

1. Generate EdDSA (ed25519) keys
   - Use SparkleÔÇÖs tools: `generate_keys` produces `ed25519` private and public keys.
   - Store the private key securely (not in the repo). Add the public key value to `Info.plist` as `SUPublicEDKey`.

2. Sign update archives
   - Use `sign_update` (or `generate_appcast`, which signs too) to produce `sparkle:edSignature` for the enclosure.

3. Code sign & notarize your app and archives
   - Sign the `.app` with your Developer ID Application certificate.
   - Notarize the `.app` and your distributable archive (`.zip` or `.dmg`).
   - Ensure SparkleÔÇÖs helper components embedded in the app are code signed appropriately; SPM embeds properly signed artifacts when linking `Sparkle`.


## 6) Release Artifacts

- Recommended: zip the `.app` into `Hardlinker-<version>-macOS.zip` using a stable, deterministic method (preserve permissions and symlinks).
- Optionally ship a `.dmg` instead of `.zip` (Sparkle supports both).
- Keep file names versioned and immutable once published.


## 7) Appcast (Feed) Generation & Hosting

1. Pick a hosting location for the appcast XML
   - Option A: Serve `appcast.xml` from this repoÔÇÖs GitHub Pages (e.g., `https://<username>.github.io/Hardlinker/appcast.xml`).
   - Option B: Serve from another HTTPS host or S3/CloudFront.

2. Generate the appcast
   - Use SparkleÔÇÖs `generate_appcast` pointing at a folder of release archives.
   - Provide the private key so entries are signed with `sparkle:edSignature`.
   - Include `sparkle:version` and `sparkle:shortVersionString` that match your app bundleÔÇÖs `CFBundleVersion` and `CFBundleShortVersionString`.
   - Provide `sparkle:releaseNotesLink` or inline HTML notes per release.
   - Optionally enable delta updates if you desire smaller downloads.

3. Host the appcast and archives
   - Commit/publish `appcast.xml` to this repo (if using Pages) or upload elsewhere over HTTPS.
   - Host downloads on GitHub Releases or your site; set enclosure `url` to the final download URL.

4. Point the app at the feed
   - Set `SUFeedURL` in `Info.plist` to the public feed URL, or set programmatically in code if you need channels.


## 8) CI/CD Automation (Recommended)

Automate building, signing, notarization, and appcast generation. Example outline using GitHub Actions:

- On tag push (e.g., `v*`):
  - Build universal `.app` (Release) with Xcode.
  - Codesign the app using secrets (Developer ID Application cert + keychain import).
  - Notarize using `xcrun notarytool` with App Store Connect API key secrets.
  - Zip the notarized `.app` to `Hardlinker-<version>-macOS.zip`.
  - Run SparkleÔÇÖs `generate_appcast` on the folder containing the new zip (and previous ones) with your Sparkle private key from repo secrets.
  - Upload the zip to GitHub Releases (attach to the tag) or your CDN.
  - Publish or commit the updated `appcast.xml` to the Pages branch or hosting bucket.
  - Optionally validate the feed URL and signature by running SparkleÔÇÖs verification tool (if available) as a final gate.


## 9) Testing the Update Flow

1. Local sanity checks
   - Run the app, open the Check for Updates menu. Confirm no errors in Console for Sparkle.
   - Test both manual check and automatic checks after launch.

2. Staging channel (optional but recommended)
   - Use a second feed URL (e.g., `appcast-staging.xml`).
   - Set it programmatically in debug/staging builds so you can test updates before production.

3. Versioning tests
   - Ensure `CFBundleShortVersionString` and `CFBundleVersion` increment appropriately. Sparkle compares versions to decide availability.
   - Verify installation path (typically `/Applications`) to allow inÔÇĹplace updates without admin prompts when possible.

4. Rollback plan
   - If a bad update ships, replace the appcast item to point back to a previous stable build or publish a hotfix with a higher version.


## 10) Security & Privacy Considerations

- Always serve appcast and downloads over HTTPS.
- Keep the Sparkle private key out of the repo; store in CI secrets or a secure keystore.
- Embed only the public key in `Info.plist`.
- Ensure your appÔÇÖs hardened runtime and sandbox entitlements are minimal and correct.
- Consider disabling system profiling (`SUEnableSystemProfiling` false) unless you explicitly want it and disclose it in privacy docs.
